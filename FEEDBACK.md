# Uniswap track feedback (Earnlab / EarnYld)

Our agents package treats Uniswap v4 as the primary signal for autonomous LP work: **`UniswapV4Scanner`** walks configured chains with viem against each chain’s **`PoolManager`**, reads pool state, enriches with DefiLlama (and optional The Graph when a key is set), and ranks pools for fee yield and risk-adjusted metrics. We ship **`@uniswap/sdk-core`**, **`@uniswap/v4-sdk`**, **`@uniswap/v3-sdk`** (shared tick math with v4), and **`@uniswap/universal-router-sdk`** in TypeScript. The hardest engineering work was keeping JSON-safe **`V4PoolKeyJson`** in the API contract aligned with the scanner’s pool keys so **`ExecutionPreflight`** and **`SlippageGuard`** can run the **v4 Quoter** only after a fresh scan supplies a full key.

Hooked pools are first-class in the product: we decode which v4 callbacks are active using **`hookFlagIndex`** from **`@uniswap/v4-sdk`** over the hooks address (`decodeHookFlags` in the scanner), and the dashboard surfaces that for operators. Flag decoding is elegant for labeling, but it does not fully describe economic behavior—hooked pools can still change fees, curves, or deltas in ways our scorecards must treat conservatively. Sponsor-side, we would value more TypeScript-oriented guidance on going from “flags from the address” to safe assumptions for ranking, simulation, and execution preflight for teams that consume hooks but do not author them.

We document the official **[uniswap-ai](https://github.com/Uniswap/uniswap-ai)** skills for contributors (`npx skills add Uniswap/uniswap-ai` in our README): they helped our team ground v4 security and swap-path discussions in the IDE, not as a runtime dependency inside the agent binary. A tighter loop of **agent-shaped** examples—viem + Quoter preflight + multi-chain pool keys shared between a Next.js UI and a Node service—would shorten the path from “skills for the coding agent” to the kind of production scanner and portfolio code Earnlab ships.
