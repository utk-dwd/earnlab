# KeeperHub track feedback (Earnlab / EarnYld)

In Earnlab / EarnYld, KeeperHub sits behind an explicit execution matrix: default `EXECUTION_MODE=paper` keeps portfolio and decision logic running with simulated accounting, while `live` turns on the path where `PortfolioManager` can drive real rebalances through our `KeeperHubClient`. That client uses the official `@modelcontextprotocol/sdk` with either `KEEPERHUB_MCP_URL` (SSE) or `KEEPERHUB_MCP_COMMAND` (stdio), calls MCP tools such as `execute_rebalance` with `AgentRegistry` / `KEEPERHUB_REGISTRY` context, and supports `KEEPERHUB_DRY_RUN=true` so demos and CI never need secrets. That shape (paper sandbox, then optional MCP-backed settlement) matches how we pitch Safe-Mode before production execution; clearer KeeperHub-side docs on the minimal env surface for Galileo plus registry would help other teams reproduce the same story faster.

Deterministic behavior for us means: the LLM proposes and critiques, but TypeScript owns registry addresses, slippage bps, and the rebalance payload. We still saw friction where a simulation or plan looked fine while pool state moved before execution, so our own v4 Quoter preflight (`quoteSwapLegForNotionalUsd` / `SlippageGuard`) and scanner freshness gates matter as much as the automation layer. It would help if MCP tool schemas exposed richer, structured errors (simulation versus chain mismatch versus slippage) and optional stricter pre-flight parameters so agents can fail closed without parsing free-text tool content.

For TypeScript MCP authors, we would like version-pinned recipes next to KeeperHub examples: SSE versus stdio tradeoffs, expected `callTool` result shapes (we parse `content[0].text` for tx hashes), and guidance when moving between major `@modelcontextprotocol/sdk` releases. A short happy-path repo that wires one `execute_rebalance` from a Node 20 agent with dry-run and live toggles would complement abstract protocol docs and reduce time-to-first successful hackathon integration.
